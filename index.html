今天在网上看到一题面试题，觉得挺有意思的。题目描述如下：

实现一个LazyMan，可以按照以下方式调用:

LazyMan(“Hank”)输出:

Hi! This is Hank!

LazyMan(“Hank”).sleep(10).eat(“dinner”)输出

Hi! This is Hank!

//等待10秒..

Wake up after 10

Eat dinner~

LazyMan(“Hank”).eat(“dinner”).eat(“supper”)输出

Hi This is Hank!

Eat dinner~

Eat supper~

LazyMan(“Hank”).sleepFirst(5).eat(“supper”)输出

//等待5秒

Wake up after 5

Hi This is Hank!

嗯！Math 这些都未定义！我在其他 js 文件里也做了相似的测试，都是这样的结果。说明所有 JS 文件被加载运行了两次！而且第一次运行的时候，执行环境是大家所熟知的，而第二次加载运行的时候，执行环境里竟然找不到 Math 等对象！我试图在测试代码里的 console.log('Math is ' + typeof Math + '\n' + 'angular is ' + typeof angular);前面打个断点，想分析第二次加载到底是怎么回事，可是这个断点只有在第一次加载的时候能卡住起到效果，第二次加载的时候没有任何效果。

在调试的过程中，本想随便点点看看别的功能是否能正常使用的，想换个突破口。在这个过程中，发现在别的页面(修改密码页面)，先手动点一下 input 框，让其获得焦点，一旦这里的 input 框获得焦点成功之后，其他地方原本不能获取焦点的输入框都能获取焦点输入了(手动黑人问号脸…)。于是就想，要不弄一个隐藏的输入框，每次进入页面的时候，先让其自动获得焦点不就可以解决这个 IE 的迷之问题了？于是在 index.html 里多了这么一段恶心的脚本：

定时 4 秒是因为要在它第二遍 js 加载完成之后再执行这段代码才有效，也就是说，如果在 4 秒内还没加载完的话，依然会无法输入。这样表面上看上去是莫名其妙的解决了这个浏览器兼容性问题了，但我实在是受不了这样恶心的代码，我觉得就算 IE 恶心，很弱，也不至于会出现加载两次 JS 的这么明显而低级的 bug 的，问题更有可能是出现自己的代码里，我必须得看看到底这个第二次加载是怎么回事，一定会有更好的办法解决这个问题的！于是我开始排除一些不必要的干扰因素，慢慢的试图准确的定位一下问题，我先把所有 index.html 引用的 js注释掉，只留下 vendor 目录下的和一个登录页面需要 js,然后惊奇的发现，这个 bug 消失了！这说明问题就是由于我自己写的，刚刚注释掉的一些 js 文件里的代码引起的。看到这里，我知道这个问题很快就能探个究竟，得到解决了。进一步的，用二分法重复几下，很快就找到了引起这个问题的罪魁祸首是 filePreview.js 这个文件。这是个文件在线预览模块，而这个模块里有用到 iframe 标签，所以，基本上可以断定，这个奇葩的第二次加载 js 文件就是这个 iframe 引起的。google 了一下 iframe 的相关问题，stackoverflow 上看到有人说 iframe 的 sandbox 属性可以做一些安全限制：

我试了一下，把 filePreview directive 里的 iframe 标签加了个 sandbox='' :

<iframe ng-if="file_type != 'default' && file_type!= 'image'" src="result_file_url" sandbox=''></iframe>

嗯，果然已经没有这些 JS 错误了,可又发现，这样一来，文件预览功能又不行了，这个 iframe 加载不了内容了，继续看相关的文档，iframe 元素, 和 security 属性，security 属性说明如下：

最后，iframe 标签如下：

<iframe ng-if="file_type != 'default' && file_type!= 'image'" src="result_file_url" security="restricted"></iframe>

不再报 JS 错误，文件预览功能也不受影响，至此，也算是解决这个问题了。从发现问题，不断猜想和去验证猜想，debug 前前后后花了我一天的时间，最后解决问题仅仅是加了个 security="restricted" ，哎，真的是有点哭笑不得的样子。但程序员嘛，总是在不断解决问题中总结，成长。最后如果能顺利解决问题，自然会多一分成就感的喜悦，最后就算未能解决问题，在搜索解决相关问题的过程中，也会阅读到很多相关文档，学到很多知识，过程自然也是愉悦的。最后我想说，以代码量来衡量一个程序员的能力与价值的话，那纯属扯淡！哈哈！
